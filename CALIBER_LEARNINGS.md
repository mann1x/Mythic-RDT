# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern:project]** When checkpoints from a partial pod run land at `checkpoints/<name>_ckptNNN/` (loose layout) rather than the trainer's expected `checkpoints/<name>/checkpoint-NNN/` subdirectory, restructure with `mkdir -p checkpoints/<name>/checkpoint-NNN && mv checkpoints/<name>_ckptNNN/* checkpoints/<name>/checkpoint-NNN/ && rmdir checkpoints/<name>_ckptNNN` BEFORE pointing `--init-from-checkpoint checkpoints/<name>` at it. The loader resolves the latest `checkpoint-*` subdir under the given root; loose files at the root are silently ignored and you get `loaded=0`.
- **[gotcha:project]** `scripts/finetune_phase1.py` rejects any `--output-dir` that starts with `/tmp` at startup with `ERROR: --output-dir under /tmp is forbidden (memory rule).` (parent project tmpfs rule). Use `/workspace/mythic-rdt/checkpoints/...` on the pod or `checkpoints/...` on solidPC — never `/tmp/<run>`.
- **[gotcha:project]** Multi-source `scp` with one destination directory puts ALL listed sources into that directory regardless of their original parent paths. When syncing files that map to different pod-side directories (e.g. `src/mythic_rdt/training/trainer.py` AND `scripts/finetune_phase1.py`), use ONE `scp` per destination — chaining them in a single command silently misplaces files and the misplaced copy can shadow the real one at import time.
- **[env:project]** The `mythic-rdt` conda env on solidPC ships no `pytest` — `python -m pytest` errors with `No module named pytest`. Run unit tests with the stdlib runner instead: `PYTHONPATH=src conda run -n mythic-rdt python -m unittest tests.test_<name> -v`. The `PYTHONPATH=src` prefix is mandatory because the package isn't installed in this env.
- **[pattern:project]** `_load_trainable_state` in `src/mythic_rdt/training/trainer.py` is STRICT by default — it raises `CheckpointShapeMismatchError` on T-axis shrinkage (the bug-050 trap: ckpt T=4 → wrapper T=1 silently dropped tensors). Override only with the env var `MYTHIC_LOAD_LENIENT=1`, which prints a warning at trainer init. Don't set this for any production-bound run; only for short forensic reproductions.

Ignore: `.wolf/daemon.log`, `.wolf/token-ledger.json`, `.wolf/cron-state.json`, `.wolf/hooks/_session.json`, `.wolf/hooks/_session.json.*.tmp`, `graphify-out/`. Track: `.wolf/anatomy.md`, `cerebrum.md`, `memory.md`, `buglog.json`, `OPENWOLF.md`, `identity.md`, `config.json`, `suggestions.json`. - **[pattern:project]** Before staging untracked OpenWolf/eval files for a commit, verify `.gitignore` decisions with `git check-ignore -v <file1> <file2> ...`. The output names the line and pattern that matched (e.g. `.gitignore:78:!eval_results/*.log eval_results/foo.log`), more trustable than re-reading patterns manually.
- **[gotcha:project]** `humaneval_smoke.py`'s `RunResult.failures` list is CAPPED at 5 entries (`if len(failures) < 5: failures.append(...)` at the LCB and HE call sites). When auditing whether wrapper-vs-base divergence is shared-failure or new-failure on a 30-problem eval, the capped list looks like only 5 fails total, but `n_total - n_pass - len(failures)` reveals the rest are unaccounted (e.g. v6A T=1 LCB: n_pass=3, n_total=30, failures=5 → 22 more fails hidden). Add a `per_problem` field to `RunResult` (`{task_id, passed, reason}` for every problem) so post-hoc set-diff (`base_pass_wrap_fail = wrap_fails - base_fails`) is exhaustive, not capped.
- **[correction:project]** Files in `src/generated/` are auto-generated — never edit them directly
- **[pattern:project]** `_load_trainable_state` in `src/mythic_rdt/training/trainer.py` is STRICT by default — it raises `CheckpointShapeMismatchError` on T-axis shrinkage (the bug-050 trap: ckpt T=4 → wrapper T=1 silently dropped tensors). Override only with the env var `MYTHIC_LOAD_LENIENT=1`, which prints a warning at trainer init: `[trainer] WARNING: MYTHIC_LOAD_LENIENT=1 set — _load_trainable_state is running in lenient mode. Silent shape mismatches WILL be tolerated.` Don't set this for any production-bound run; only for short forensic reproductions where the shrinkage is intentional.
- **[gotcha:project]** Concurrent base + wrapper on the same 24 GB 3090 OOMs at bs=4 on DS-V2-Lite (NF4 base ~10 GB + wrapper extras + a 1.6 GB fp32 eager-attention scores tensor = >23 GB peak). Halving to bs=2 doesn't help — peak is dominated by the attention-scores allocation, not the batch. Either run sequentially (Phase A: base alone → save tokens to disk → free; Phase B: wrapper alone → save → free; Phase C: diff from disk) like `humaneval_smoke.py` does, or switch `attn_implementation="flash_attention_2"` to drop the scores tensor entirely.
- **[gotcha:project]** DeepSeek-V2's `ATTENTION_CLASSES` registry only contains `eager` and `flash_attention_2` — there is NO SDPA implementation. `attn_implementation="sdpa"` will silently fall back or KeyError. To gain any attention-time/memory savings on DS-V2-Lite you MUST install flash-attn; SDPA is not an option.
- **[gotcha:project]** `pip install --no-build-isolation flash-attn` in the `mythic-rdt` env fails because system `nvcc` at `/usr/local/cuda` symlinks to **CUDA 13.2** while torch is built against **CUDA 12.6**. Multiple CUDA toolchains coexist under `/usr/local/cuda-12.4`, `/usr/local/cuda-12.6`, `/usr/local/cuda-13.2` — point the build at the matching one with `CUDA_HOME=/usr/local/cuda-12.6 PATH=/usr/local/cuda-12.6/bin:$PATH pip install ...` OR (faster) install a prebuilt wheel from `https://github.com/Dao-AILab/flash-attention/releases/tag/v2.8.3` — match `cu12torch2.6cxx11abi<TRUE|FALSE>-cp312` to `python -c "import torch; print(torch._C._GLIBCXX_USE_CXX11_ABI, torch.__version__)"`. The `fa4-v4.0.0.beta*` tags are FA4 betas, not FA2 — use the `v2.x.x` tags for FA2.
- **[gotcha:project]** solidPC runs Debian 11 with **glibc 2.31** (`ldd --version`). Some prebuilt FA2 wheels (esp. cu12torch2.6cxx11abiTRUE) link against glibc symbols newer than 2.31 and import-fail at runtime. If a wheel installs cleanly but `import flash_attn` errors with `GLIBC_2.32 not found` or similar, uninstall and try the `cxx11abiFALSE` variant or build from source with `CUDA_HOME=/usr/local/cuda-12.6`.
- **[pattern:project]** When a probe that needs A/B comparison of two large models can't fit both on the GPU, restructure as **sequential phases** that persist intermediate state to disk (`/tmp/probe_*.pt` is OK for ephemeral 30-token-list payloads, NOT for weights/intermediates per project tmpfs rule): Phase A loads model 1, generates, `torch.save(tokens, path)`, deletes the model, runs `gc.collect(); torch.cuda.empty_cache()`. Phase B loads model 2, generates, saves. Phase C loads both token lists from disk and diffs. Verify post-free with `torch.cuda.mem_get_info()` — if free GPU isn't ~full base before Phase B, something is leaking.
- **[gotcha:project]** Long-running tasks launched via the Bash tool's `run_in_background` mode write stdout to `/tmp/claude-0/.../tasks/<taskId>.output`. Use `tail -50 <path>` to inspect mid-run; `TaskOutput` with `block=true` will time out and return only partial buffered text on still-running tasks. The `bzk10owhz.output` reading pattern in this session is the correct way to read failed `pip install` traces that don't surface via `TaskOutput`.
- **[gotcha:project]** Mythic-RDT `modeling.py` has TWO recurrence loops in `forward()`: a `use_cache=True` inline branch and a `use_cache=False` branch that delegates to `_loop_step`. They are NOT a unified code path. ANY wrapper-semantics change (v6E `first_iter_identity`, force_gate_zero, force_bypass, layerscale clamps, `set_loop_t(-1)` short-circuits) MUST be mirrored in BOTH branches. HF `GenerationMixin.generate()` always passes `use_cache=True`, so the inference path is the inline branch — features only added to `_loop_step` will silently no-op at inference and diverge from base. After ANY recurrence-path edit, run `scripts/_probe_wrapper_generate_lcb.py` (sequential phases, 30-problem batched-padded-384-token) to verify byte-identity at T=1 before trusting any v6E claim. Long-term: consolidate the two branches into one helper that takes a cache + checkpoint flag.
- **[gotcha:project]** When an LCB/HumanEval smoke shows wrapper underperforming base at T=1 with `first_iter_identity=True`, do NOT assume the model is broken or that LoRA[0] needs re-zeroing. The bug is likely the inference-path divergence above (use_cache=True branch missing v6E logic) — old v6A/v6E checkpoints were trained correctly but eval'd through a buggy inference path. Re-eval with the fix BEFORE assuming any ckpt is unsalvageable. The diagnostic chain that catches this: (1) per-problem dump → most failures are base-passes, (2) `_probe_wrapper_generate_lcb.py` shows `first_div >= 0` on all problems → wrapper diverges at very first token → inference-path bug, not training-path bug.
- **[fix:project]** When a per-problem audit is needed and `RunResult.failures` is capped at 5 entries, do NOT increase the cap — the cap was added intentionally to keep the JSON small. Instead add a `per_problem` field to `RunResult` that records `{task_id, passed, reason}` for every problem (small, ~30 entries × 30 chars). Post-hoc set-diff `wrap_fails - base_fails` then reveals exactly which problems newly broke under the wrapper. The capped-failures approach made it impossible to diagnose v6E divergence on 2026-04-29.
- **[gotcha:project]** FA2 cannot run on solidPC — Debian 11 host glibc is 2.31 but the conda env's `libstdc++.so.6.0.34` references `__libc_single_threaded@GLIBC_2.32`, which the build links into `flash_attn_2_cuda.cpython-312...so` regardless of `LDFLAGS="-L/usr/lib/x86_64-linux-gnu -Wl,-rpath,..."`. `LDFLAGS` only affects link-time *resolution*, not which symbol versions get baked in. Even after a clean wheel rebuild with system libstdc++ paths, `nm -u` still shows the GLIBC_2.32 symbol → `import flash_attn` fails with the same error. Verify with `nm -u <so> | grep GLIBC_2.32` BEFORE assuming a fresh build will solve it. Real fixes (untried as of 2026-04-29): run in Debian 12+/Ubuntu 22.04+ container, or `conda install -c conda-forge libstdcxx-ng=11` to downgrade conda's libstdc++. Until then: stick with `attn_implementation="eager"` locally and use the pod (Ubuntu 22, glibc 2.35) for any FA2-dependent training.
- **[gotcha:project]** A broken FA2 install in the conda env breaks ALL transformers code paths that import flash_attn anywhere in their resolution — even if the script never asked for FA2 explicitly. Symptom: `_probe_wrapper_generate_lcb.py` (which loads with `attn_implementation="eager"`) crashes with `ImportError: ... GLIBC_2.32 not found ... flash_attn_2_cuda...`. Fix: `conda run -n mythic-rdt pip uninstall -y flash-attn` and rerun. After uninstall, `python -c "import transformers; print(transformers.__version__)"` should succeed cleanly. Always uninstall a broken FA2 wheel before launching unrelated long-running probes.
- **[gotcha:project]** When the wrapper KV cache is split per-iteration (`dict[int, DynamicCache]`) and HF generate continuation-step passes a single seq-length value through `prepare_inputs_for_generation`, a pre-existing `_MythicCacheView` returned to HF MUST delegate `get_seq_length(layer_idx=0)` to `iter_caches[0]` (the t=0 sub-cache), not to a top-level fictitious cache. Otherwise HF reports past_len=0 on every continuation and re-encodes the prompt every step — silently quadratic generation cost masked as "slow".
- **[gotcha:project]** Stale `.pyc` files in `src/mythic_rdt/__pycache__/modeling.cpython-312.pyc` can shadow edits to `modeling.py` even when `mtime` of the `.py` is newer — Python's import system uses the `.pyc` if it judges it valid. ALWAYS launch any probe/smoke that depends on a recurrence-path edit with `PYTHONDONTWRITEBYTECODE=1` in the env. Verify the source actually contains the new logic with a one-liner: `python -c "src=open('.../modeling.py').read(); print('NEW_TOKEN' in src)"` BEFORE concluding 'fix didn't work'. The 2026-04-29 v6E fix was nearly mis-diagnosed because Phase B output predated the modeling.py edit (compare `stat -c %y modeling.py` to log birth time).
- **[pattern:project]** When debugging wrapper-vs-base token divergence, save raw token ID lists (not decoded strings) with `torch.save(list_of_lists, path)` and diff in a separate Python invocation: `min(len(bt), len(wt))` then `next((j for j in range(n) if bt[j] != wt[j]), -1)` for `first_div`, plus `n_match = sum(1 for j in range(n) if bt[j] == wt[j])`. `first_div=-1` AND `len(bt)==len(wt)` is the only signal of byte-identity. Do not rely on text decoding — the same string can decode from different token sequences via merge/split ambiguity.

- **[gotcha:project]** `pip install --no-build-isolation <pkg>` reuses the cached wheel by `(pkg, version)` — NOT by build env vars. Setting new `LDFLAGS`/`CUDA_HOME`/`TORCH_CUDA_ARCH_LIST` and re-running pip will silently install the previous build's cached wheel without recompiling, even though pip's output looks like a fresh build. Verify with `pip cache list <pkg>` (look at the wheel's SHA — identical SHA across two "successful" builds = the cache was reused). To force a real rebuild: `conda run -n <env> pip cache remove <pkg>` BEFORE the install, OR add `--no-cache-dir` to the install command. Burnt 2026-04-29 retrying flash-attn three times with different LDFLAGS getting the same broken `.so`.
- **[gotcha:project]** `pip install --no-build-isolation` defaults `TMPDIR` to `/tmp`, which is tmpfs (RAM-backed). When the pip cache lives on persistent disk (`/srv/dev-disk-by-label-opt/.cache/pip/wheels/...`), pip's final "rename wheel into cache" step fails with `OSError: [Errno 18] Invalid cross-device link` — and the misleading symptom is that the build looks like it succeeded ("Successfully built ...") then fails at the last second. Fix: `TMPDIR=/srv/dev-disk-by-label-opt/tmp_<scratch> pip install ...` so TMPDIR is on the same filesystem as the pip cache. This also obeys the project rule "never write large files to /tmp". Applies to any pip wheel-build of a >100 MB package (flash-attn, bitsandbytes from source, vllm, etc.).
- **[pattern:project]** When concurrent base+wrapper OOMs and you suspect another process is squatting on the GPU, identify the holder with `nvidia-smi --query-compute-apps=pid,used_memory,process_name --format=csv` then `cat /proc/<pid>/cmdline | tr '\0' ' '` and `ps -fp <pid>`. The cmdline gives you the actual launch command (often a Docker container's `python3 -u /app/main_em.py` for unrelated services on solidPC). Don't assume the OOM is your script — verify the GPU is actually free with `nvidia-smi --query-gpu=memory.free --format=csv,noheader` before retrying.
- **[pattern:project]** Use the `gh api` CLI (NOT WebFetch) for GitHub release asset enumeration: `gh api 'repos/<owner>/<repo>/releases?per_page=15' --jq '.[].tag_name'` to list recent tags, then `gh api 'repos/<owner>/<repo>/releases/tags/<tag>' --jq '.assets[].name' | grep <pattern>` to filter wheel filenames. The latest release is often a beta on a different major version (e.g. flash-attention's `latest` was `fa4-v4.0.0.beta10`, but FA2 stable was at `v2.8.3`) — never trust `releases/latest` for stable wheel selection.
