# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha:project]** HF Trainer `--resume` restores AdamW optimizer state (`exp_avg`/`exp_avg_sq`) sized to the prior trainable param shapes. If the new run changes a LoRA-bearing dim such as `--max-loop-iters` (e.g., 1 → 4 expands LoRA T-axis from `[1,8,2048]` to `[4,8,2048]`), restore crashes inside `_multi_tensor_adamw` with `RuntimeError: output with shape [1, 8, 2048] doesn't match the broadcast shape [4, 8, 2048]`. Use `--init-from-checkpoint <prev>` instead — it loads only `mythic_rdt_trainable.pt` (with automatic T-axis expansion: leading slices = checkpoint values, new slices = fresh init) and starts a fresh optimizer/scheduler/global_step. Curriculum start_steps are then absolute from 0, not offset.
- **[pattern:project]** `_load_trainable_state` in `src/mythic_rdt/training/trainer.py` performs T-axis expansion on shape mismatch: if checkpoint dim 0 ≤ target dim 0 and all trailing dims match, it copies into the leading slices. Confirm with the log line `[ft] init-from-checkpoint: loaded=N missing=0 unexpected=0` — for v3-T1→v4 (T=1→T=4) expect loaded=80 (LoRA-A/B for q/o on each of 19 block layers + gate.bias + layerscale.scale + lti.B.weight + lti.log_a). At init step KL(force_bypass||wrapper) ≈ 1e-3 nats/token, confirming clean inheritance.
- **[gotcha:project]** Editing `scripts/finetune_phase1.py` to add a new flag (e.g., `--curriculum-style v4-anchored`, new `--gate-init-bias`/`--layerscale-clamp-max`/`--kl-anchor-alpha`/`--kl-anchor-every`) requires three coordinated edits: (1) `argparse.add_argument` in the script, (2) thread the value into `MythicRDTTrainer(...)` / `MythicRDTDeepseekV2Config(...)` / curriculum factory, (3) register any new curriculum function in `mythic_rdt/training/curriculum.py` `__all__`, AND `rsync` to the pod before relaunching. Cheap forgetting trap: launch reads the new flag fine but config/trainer defaults silently win. Verify with a 3-step smoke run + `grep` on the pod-side files for the new symbol after each rsync.
- **[pattern:project]** When watching a long remote training/eval to completion, prefer a single `Monitor` poll loop that ssh-tails log size + `pgrep` and emits one event per ~500B-2.5KB log growth, then a final "EXITED" event. Cheaper on context than polling Bash every minute, and the exit event is the trigger for downstream actions (rsync results, launch eval). Throttle the size-growth threshold (>500B for short jobs, >2KB for multi-hour runs) so each notification represents real progress, not buffer flushes.
- **[pattern:project]** KL-to-base anchor in `MythicRDTTrainer.compute_loss`: when `kl_anchor_alpha > 0` and `global_step % kl_anchor_every == 0`, run a second `model(..., T=1, force_bypass=True, return_dict=True)` under `no_grad()` to get base logits, then `loss += alpha * F.kl_div(log_softmax(w_logits), softmax(b_logits), reduction='batchmean') / w_logits.size(1)`. Same wrapper instance — no second model resident. Divide by seq_len so KL is per-token (same scale as CE). Use `force_bypass=True` (not `T=1` alone) so the recurrent path is fully zeroed. Log `kl_anchor` scalar to wandb each anchored step. v4-anchored defaults: `alpha=0.05`, `every=8` (~6% extra cost).
- **[env:project]** solidPC RTX 3090 (24 GB) is shared with persistent GPU consumers (Plex/HW transcode, ollama runner ~4.6 GB, fan2go, em.py ~0.5 GB) totaling ~5+ GB before any training. A DS-Coder-V2-Lite NF4 load (~10 GB weights + activations) plus the wrapper can OOM if launched without `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True`. Before any local 3090 NF4 smoke: run `nvidia-smi --query-gpu=memory.free --format=csv,noheader` and `fuser -v /dev/nvidia*` to enumerate other holders; expect ~19 GB free, leaving headroom only for batch_size=1-2 at seq≤1024. For larger configs, prefer the rented A6000 pod.
- **[gotcha:project]** Backgrounding via `nohup ... & disown` from the Bash tool with `run_in_background=true` returns an empty `output` from `TaskOutput` even after the launch shell exits cleanly — the file descriptors are detached before the harness can capture stdout. Verify the launch by reading the redirected log file (`rtk read <log> --tail-lines N`) and `pgrep -af <script>` rather than trusting an empty TaskOutput. Reference: events 7-8 and 15-16 both reported empty output for successful `humaneval_smoke.py` launches.
- **[gotcha:project]** Local solidPC env name is `mythic-rdt` (`source /root/anaconda3/etc/profile.d/conda.sh && conda activate mythic-rdt`), which ships `transformers==4.46.3` + `torch==2.6.0+cu126` + `bitsandbytes==0.49.2`. This is the sidecar env required for DS-Coder-V2-Lite (broken under transformers 5.x — see existing memory). Do NOT activate `lightseek` for any DS-Coder modeling — `lightseek` has transformers 5.5.0 and will fail at base load. Pod-side equivalent is `/workspace/venv-tf4/bin/activate`.
- **[pattern:project]** When checkpoints from a partial pod run land at `checkpoints/<name>_ckptNNN/` (loose layout) rather than the trainer's expected `checkpoints/<name>/checkpoint-NNN/` subdirectory, restructure with `mkdir -p checkpoints/<name>/checkpoint-NNN && mv checkpoints/<name>_ckptNNN/* checkpoints/<name>/checkpoint-NNN/ && rmdir checkpoints/<name>_ckptNNN` BEFORE pointing `--init-from-checkpoint checkpoints/<name>` at it. The loader resolves the latest `checkpoint-*` subdir under the given root; loose files at the root are ignored silently and you get `loaded=0`.
- **[gotcha:project]** `scripts/finetune_phase1.py` rejects any `--output-dir` that starts with `/tmp` at startup with `ERROR: --output-dir under /tmp is forbidden (memory rule).` (parent project tmpfs rule). Use `/workspace/mythic-rdt/checkpoints/...` on the pod or `checkpoints/...` on solidPC — never `/tmp/<run>`. Reference: 2026-04-28 v5 probe died at launch before any model load with this message.
- **[gotcha:project]** Multi-source `scp` with one destination directory puts ALL listed sources into that directory regardless of their original parent paths. `scp file1.py file2.py root@host:/dst/` will land both files under `/dst/`, even if file2 belongs in a different remote subdir. When syncing files that map to different pod-side directories (e.g. `src/mythic_rdt/training/trainer.py` AND `scripts/finetune_phase1.py`), use ONE `scp` per destination — chaining them in a single command silently misplaces files and the misplaced copy can shadow the real one at import time.
- **[env:project]** The `mythic-rdt` conda env on solidPC ships no `pytest` — `python -m pytest` errors with `No module named pytest`. Run unit tests with the stdlib runner instead: `PYTHONPATH=src conda run -n mythic-rdt python -m unittest tests.test_<name> -v`. The `PYTHONPATH=src` prefix is mandatory because the package isn't installed in this env (no `pip install -e .` performed on solidPC).
- **[pattern:project]** `_load_trainable_state` in `src/mythic_rdt/training/trainer.py` is now STRICT by default — it raises `CheckpointShapeMismatchError` on T-axis shrinkage (the bug-050 trap: ckpt T=4 → wrapper T=1 silently dropped tensors). Override only with the env var `MYTHIC_LOAD_LENIENT=1`, which prints a warning at trainer init: `[trainer] WARNING: MYTHIC_LOAD_LENIENT=1 set — _load_trainable_state is running in lenient mode. Silent shape mismatches WILL be tolerated.` Don't set this for any production-bound run; only for short forensic reproductions where the shrinkage is intentional.
- **[convention:project]** `.gitignore` for OpenWolf-managed projects must split runtime state from authored content. Ignore: `.wolf/daemon.log`, `.wolf/token-ledger.json`, `.wolf/cron-state.json`, `.wolf/hooks/_session.json`, `.wolf/hooks/_session.json.*.tmp`, `graphify-out/`. - **[pattern:project]** Before staging untracked OpenWolf/eval files for a commit, verify `.gitignore` decisions with `git check-ignore -v <file1> <file2> ...`. The output names the line and pattern that matched (e.g. `.gitignore:78:!eval_results/*.log eval_results/foo.log`), which is more trustable than re-reading the patterns manually. Use this whenever you've added new ignore rules or are unsure why a file is/isn't tracked.
- **[pattern:project]** When wiring v5-style dual-T training, the trainer's `compute_loss` activates dual-T iff `margin_alpha > 0` OR `distill_alpha > 0` (otherwise single-T curriculum path). The dual-T branch does TWO `model(...)` calls per microbatch (at `t_lo` and `t_hi`) — wall-clock per step roughly doubles vs the v4 single-T path. Plan ETAs accordingly: v5 probe @ 1×3090 NF4 ≈ 120 s/step → 200 steps ≈ 7 h. Logged scalars: `margin_loss`, `distill_loss`, `ce_lo_T`, `ce_hi_T`, `ce_gap`, `T_lo`, `T_hi`, `kl_anchor` (when also enabled). Watch `ce_gap` trend — positive at init means high-T is currently worse and the margin term is doing its job.
- **[gotcha:project]** Parsing pod-side JSON via `ssh ... "... | python3 -c 'script'"` is fragile when the script uses f-strings with embedded `"` (e.g. `f"{d[\"args\"][k]}"`) — bash + ssh + python escape layers collide and produce `SyntaxError: unexpected character after line continuation character`. Reference: 2026-04-28 event parsing `v5_lcb10.json`. Working pattern: `scp -P <port> root@<pod>:/workspace/.../*.json eval_results/`, then run a clean Python heredoc locally with no escape gymnastics. Cheaper to debug and the JSON archives the pod artifact alongside the local result.
- **[pattern:project]** When mirroring pod training checkpoints to solidPC for inference / smoke-eval (NOT training resumption), use `rsync --exclude='optimizer.pt' -avz -e 'ssh -p <port>' root@<pod>:.../checkpoints/<run>/ checkpoints/<run>/`. The optimizer state is ~2× the trainable state per ckpt (e.g. v5 ckpt: 22 MB optimizer.pt vs 11 MB mythic_rdt_trainable.pt) and is only needed if you'll `--resume` training on the same machine. `--init-from-checkpoint` and smoke-eval load only the trainable state.
- **[convention:project]** Pod orchestration scripts (`run_v5.sh`, `run_overnight.sh`, etc.) emit `[<run>] PHASE 1: ...`, `[<run>] PHASE 2a: ...`, `[<run>] PHASE 2b: ...`, `[<run>] HH:MM ALL DONE` markers on stdout. Persistent `Monitor` tasks watching the run log can grep `^\[<run>\] PHASE` + `ALL DONE` for cheap phase-transition events. Keep this convention for any multi-step pod run (train + eval-A + eval-B) — it's the contract the monitor relies on.
- **[gotcha:project]** Long-running ssh commands (e.g., `pkill -KILL ...; sleep 2; pgrep -af ...; nvidia-smi`) can hang silently and return empty responses — Bash tool reports success but `stdout=""`. Add `-o ConnectTimeout=15` (or `-o ConnectTimeout=10`) to the ssh invocation to surface the actual issue; on retry the same command typically succeeds.
- **[gotcha:project]** Stale `.pyc` shadowing on the pod: after `scp`'ing edited `src/mythic_rdt/*.py` to the pod, the existing `.pyc` in `src/mythic_rdt/__pycache__/` can shadow the patched `.py` even when timestamps look fresh. Empirically (bug-049): same patched `modeling.py` gave wrapper T=1 LCB-medium 0% with stale pyc, then 30% (matching base) once `__pycache__` was purged. ALWAYS prefix any pod-side launch that imports a patched module with `rm -rf /workspace/mythic-rdt/src/mythic_rdt/__pycache__` (or use `PYTHONDONTWRITEBYTECODE=1`). Same general gotcha as the HF dynamic-module cache for `base/<MODEL>/modeling_*.py`, but for in-tree wrapper code.
- **[gotcha:project]** `scripts/humaneval_smoke.py` does NOT accept `--max-loop-iters` — only `--T-values`. The wrapper's `max_loop_iters` is set on the saved config (or via the smoke script's own config-build site), not by a CLI override. Passing `--max-loop-iters N` errors with `unrecognized arguments`. Reference: 2026-04-28 probe 1 first launch died at `argparse` before any model load with this exact message.
- **[convention:project]** Adding a new config field to `MythicRDTDeepseekV2Config` (e.g., `first_iter_identity`, `--gate-init-bias`, `--layerscale-clamp-max`, `--kl-anchor-alpha`) requires FOUR coordinated edits, not three: (1) `__init__` arg + `self.x = bool(x)` assignment in `src/mythic_rdt/configuration.py`, (2) `argparse.add_argument` + threading into `cfg = MythicRDTDeepseekV2Config(...)` in `scripts/finetune_phase1.py`, (3) same in `scripts/humaneval_smoke.py` (smoke also builds its own cfg, easily forgotten), (4) any modeling.py read site (e.g. `getattr(self.config, "first_iter_identity", False)`) — and `rsync` to the pod before relaunching. Cheap forgetting trap: launch reads the new flag fine but config defaults silently win on the side script you missed. Verify with `grep -n` on each of the four files for the new symbol after each edit.
- **[pattern:project]** For real-time streaming of pod-side log events to the chat, use `Monitor` with `ssh ... "tail -n 0 -F <log>" | grep -E --line-buffered "<terminal-events>"` rather than periodic `tail -200 | grep` polling inside a `while true` loop. The polling form re-tails the same lines repeatedly (causing duplicate events) and hides per-line latency; `tail -F | grep --line-buffered` emits one notification per matching line as it lands. Always use `--line-buffered` in the grep — without it, pipe buffering delays events by minutes. Always use `tail -n 0` (start at EOF) to avoid replaying historical content.
- **[gotcha:project]** Chaining a remote ssh launch + a local `nohup ... & disown` in a single `Bash run_in_background=true` invocation reliably hangs at the local-side `nohup` step — `TaskOutput` returns the ssh output but never advances past `=== POD ===` marker. Symptom: `retrieval_status: timeout` indefinitely; the local launch never actually fires. Fix: split into TWO separate Bash calls (one ssh-only, one local-only). Each `disown`'s background process inherits a clean stdin/stdout from its own bash invocation. Reference: 2026-04-28 events 4-9 — chained launch hung; split version succeeded immediately.
- **[pattern:project]** Wrapper-vs-base divergence triangulation: the decisive probe is `humaneval_smoke.py --bypass-trainables` (added 2026-04-28). It builds the wrapper, injects DepthLoRA at the same targets/rank as training, then explicitly zeros gate.bias→-10 (sigmoid≈0), LayerScale→0, lora_B→0. With LayerScale=0 (block_mode) the recurrence collapses to `h_next = block_out` so the wrapper is mathematically a no-op recurrence ≡ base — IF prelude+block+coda tiles all base layers. Run this BEFORE blaming training drift, block size, or modeling.py — it isolates plumbing from injection. Pair with a probe at a SHRUNK block (e.g. `--recurrent-block-start 10 --recurrent-block-end 14`) to rule out block-size as the failure mode. If `--bypass-trainables` ≈ base AND shrunk-block ≈ 0 % AND trained-ckpt ≈ 0 % → the failure is the recurrence injection itself at "near-identity" magnitudes, not a code bug. Memory: `project_phase1_v6_diagnosis.md`.
- **[pattern:project]** Inspecting a saved Mythic-RDT trainable checkpoint without loading the full wrapper: `torch.load('.../mythic_rdt_trainable.pt', map_location='cpu', weights_only=True)` returns a dict of named state. Bucket by suffix patterns: `lora_A` / `lora_B` (LoRA rails, expect 38 each for q/o on 19-layer block at rank 8), `gate.bias` (1 scalar), `layerscale.scale` (1 vector or per-T tensor), `lti.log_a` + `lti.B.weight` (2). Total expected: 80 keys. Mismatches expose silent T-axis truncation or missed targets before you commit GPU time on smoke/eval. Run with the `mythic-rdt` env: `PYTHONDONTWRITEBYTECODE=1 conda run -n mythic-rdt python -c '...'`.
- **[convention:project]** New config fields read from `modeling.py` MUST use `getattr(self.config, "field_name", default)` not `self.config.field_name` — direct attribute access crashes when loading legacy checkpoints (v3-T1, v4, v5) whose serialized config dict predates the field. The `getattr` idiom lets pre-existing checkpoints continue to load with the safe default while new training runs see the patched behavior. This is enforced for `first_iter_identity` by `tests/test_first_iter_identity.py::test_uses_getattr_for_safe_default`. Reference: 2026-04-28 v6A wiring.
- **[pattern:project]** Lock architectural invariants in source via `inspect.getsource()`-based unit tests instead of relying solely on behavioral regression. `tests/test_first_iter_identity.py` does `inspect.getsource(MythicRDTDeepseekV2ForCausalLM._loop_step)` then asserts the source contains `"first_iter_identity"`, `"t == 0"`, `"return block_out"`, and `'getattr(self.config, "first_iter_identity"'`. This catches a future cleanup that "removes dead code" without running the multi-hour LCB-10 smoke that would otherwise be the only signal. Add a test like this whenever a single-line invariant in the recurrence loop body is load-bearing for correctness (e.g., t=0 short-circuit, fp32 RMSNorm cast in the recurrence path, force_bypass branch).
- **[gotcha:project]** When changing a structural recurrence invariant (e.g., flipping `first_iter_identity` False→True), do NOT `--init-from-checkpoint` from a run trained under the old invariant. The previous checkpoint's LoRA-B / LTI / gate weights were optimized to compensate for the old loop body; loading them into a wrapper with the new loop body produces an inconsistent state that contradicts both objectives. Use a fresh init instead. Reference: v6A run uses no `--init-from-checkpoint` because v3-T1/v4/v5 checkpoints were trained with `first_iter_identity=False` baked into their gradients.
