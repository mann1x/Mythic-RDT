# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[gotcha:project]** The log line `[smoke] incremental JSON already at <path> — skipping legacy overwrite` is the SUCCESS signal that `IncrementalEvalState` respected existing partial state on resume — not a warning, not a bug. Do NOT delete the JSON thinking it's stale; the resume path correctly reports `N already done, 0 todo` and exits with the original pass@1 intact.
- **[gotcha:project]** Belt-and-suspenders cron sync from pod: when a system crontab line `*/25 * * * * /bin/bash <sync_script>` is already active, do NOT also schedule a Claude `CronCreate` recurring sync — they'll race on `~/pod_sync/` rsync targets and the user-fired sync requests in this session both invoke the same script directly. Single source of truth: leave the system crontab as the canonical timer; respond to user-fired sync requests by running the script ad-hoc and reporting current step from `tail -2 V6K_RUN.log`.
- **[gotcha:project]** No SSH alias `pod` exists in `~/.ssh/config` — vast.ai access requires the explicit `ssh -p 22024 -o StrictHostKeyChecking=no root@ssh6.vast.ai` form (or `-o ConnectTimeout=10` for diagnostics). Local hosts DO have aliases: `wsl`/`pandorum` (192.168.178.25), `jetnano` (192.168.178.15), `runpod-train` (ssh.runpod.io). Trying `ssh pod ...` fails with `Could not resolve hostname pod`. CAVEAT: in some shells/sessions the `pandorum` alias resolves silently empty (the `ssh pandorum '...'` call returns 0-byte stdout/stderr without the command running) — fall back to `ssh wsl '...'` which has been reliable.
- **[pattern:project]** Cache the NF4-quantized DS-Coder-V2-Lite-Instruct base ONCE via `scripts/cache_nf4_base.py --src base/DeepSeek-Coder-V2-Lite-Instruct --dst base/DeepSeek-Coder-V2-Lite-Instruct-nf4`. The script loads with `BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_quant_type='nf4', bnb_4bit_compute_dtype=bf16, bnb_4bit_use_double_quant=True)`, calls `model.save_pretrained(dst, safe_serialization=True)`, copies tokenizer + all `*.py *.json *.md` files (required so `trust_remote_code=True` works out of the cached dir). Subsequent loads point `--base` at the cached dir with `--quant none` — the saved `config.json` already has `quantization_config` baked in. Drops load+quantize wall from 10-15 min to 30-60s. Re-cache only when changing `bnb_4bit_compute_dtype` (default bf16 matches smoke's `--dtype bfloat16`).
- **[pattern:project]** Two-stage VRAM probe for tight-margin local eval on solidPC's shared 3090 (e.g. 12 GB free with ollama resident): Stage 1 = small probe (T=4 only, lcb-limit 5, batch=1) — measures peak VRAM and exits ~10 min on OOM; Stage 2 = full eval (T=1 2 4, lcb-limit 10, batch=1) ONLY if Stage 1 returned exit_code=0. `IncrementalEvalState` resumes from last task_id if mid-stage OOM kills the python. Reference: `scripts/local_v6k_ckpt200_gate.sh` — sets `PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True` to reduce fragmentation in the 12 GB window.
- **[pattern:project]** Sequential GPU-job handoff via a single Monitor (no manual coordination): `until grep -q "=== gate DONE" <log>; do sleep 30; done; sleep 15; nvidia-smi --query-gpu=memory.free --format=csv,noheader; nohup conda run --no-capture-output -n <env> python -u <next_script> > <log> 2>&1 &`. The `sleep 15` after the DONE marker lets CUDA fully release VRAM before the next job's `from_pretrained` allocator races for the same memory. Useful when a single 3090 must run an eval THEN a long preprocessing/quantization step back-to-back without user babysitting.
- **[gotcha:project]** Monitor tool bodies run in a bare shell that has NOT sourced `/root/anaconda3/etc/profile.d/conda.sh` — wrapping `conda run -n <env> python ...` or `conda activate ...` in a Monitor command fails immediately with `nohup: failed to run command 'conda': No such file or directory` and the trigger silently dies. Two fixes: (1) source the conda init script first inside the Monitor body, or (2) call the env's python directly: `/root/anaconda3/envs/mythic-rdt/bin/python -u scripts/<file>.py`. Option 2 is more portable. Same trap applies to any bash-launched background body that isn't a real interactive login shell.
- **[pattern:project]** When chaining `ckpt-N lands → wait for GPU → fire eval` triggers across multiple checkpoints concurrently, gate each new launch on a `pgrep -f 'humaneval_smoke|local_v6k_ckpt_lcb10'` busy-loop with `sleep 60; nvidia-smi --query-gpu=memory.free` BEFORE the eval kicks off. Without it, two checkpoints landing close together (e.g. 100 and 300 inside the same hour) will race for the 12 GB free window on the 3090 and the second one OOMs. The `local_v6k_ckpt_lcb10.sh CKPT_NUM` runner only handles one ckpt — concurrency is the trigger Monitor's job.

Easy isn't used in v6H/v6I/v6K/v6L downstream comparisons (saves ~35 min/run); hard is too far from current wrapper capability to be informative. - **[gotcha:project]** Self-deadlocking trigger Monitor: a Monitor body that uses `pgrep -f 'humaneval_smoke|local_v6k_ckpt_lcb10'` to wait for "another eval to finish" will match ITS OWN bash command line (which contains the literal `local_v6k_ckpt_lcb10` string from the pgrep pattern itself) and loop forever — log floods with `[ckpt300-trigger] another local eval is running, waiting 60s` even after the previous eval is long gone. Fix: narrow the pgrep to the actual python process (`pgrep -f 'python.*humaneval_smoke'`), or grep the JSON output marker instead of the script name. When you see this pattern of identical wait messages every 60s, `pgrep -af <pattern>` and look for self-matches in the Monitor's own snapshot snippet.
- **[gotcha:project]** Cached NF4 base (`base/DeepSeek-Coder-V2-Lite-Instruct-nf4/`) loads ONLY in the `mythic-rdt` conda env (transformers 4.46.3). The `lightseek/.venv` (transformers 5.5.0) crashes ~30s into the load with `NotImplementedError: "normal_kernel_cuda" not implemented for 'Byte'` inside `_init_weights` because tf 5.x re-runs init on every weight including the uint8 NF4 quant_state tensors, while tf 4.x skips init for keys present in the quantized state dict. Every local eval script that uses the cached NF4 base MUST start with `source /root/anaconda3/etc/profile.d/conda.sh && conda activate mythic-rdt` (solidPC) or `source /home/claude_test/miniconda3/etc/profile.d/conda.sh && conda activate mythic-rdt` (pandorum). Bug-073, 2026-04-30. NEVER use lightseek/.venv for cached-NF4 loads.
- **[gotcha:project]** When cloning a `local_v6X_ckpt_lcb10.sh` script for a new variant (e.g. v6L from v6K), copy the CONDA ACTIVATION BLOCK verbatim from a recent working sibling — don't write it fresh. The trap is reflexively typing `source /shared/dev/lightseek/.venv/bin/activate` when the working v6K version uses `source /root/anaconda3/etc/profile.d/conda.sh && conda activate mythic-rdt`. The crash signature is `NotImplementedError: "normal_kernel_cuda" not implemented for 'Byte'` thrown ~30s into the load, AFTER loading the cached NF4 shards — wastes a full eval slot.
- **[pattern:project]** Don't reject a v6 architecture variant on LCB-10 ckpt-trajectory alone. n=10 LCB has SE ~14pp; a +1pp wrapper-positive signal at T=4 easily flips into a -10pp "rejection" between adjacent ckpts. LCB-30 (SE ~8pp) is the authoritative metric. Concrete: v6K ckpt-400 LCB-10 reported T=4 20% (looked like REJECT vs base 30%), but full LCB-30 showed T=4 33.3% (+1 problem vs base = first wrapper-positive T=4 result across all v6{A,E,F,G,H,I,K} runs). Always confirm a kill verdict with full LCB-30 medium n=30 before designing the next variant on the assumption that this axis is dead.
- **[gotcha:project]** Eval JSON result records (`eval_results/v6X_ckptN_lcb10_local.json`) do NOT contain the actual generation strings under `generations` or `samples` — those keys return 0-length lists. Per-result schema is `{name, pass_at_1, n_pass, n_total, elapsed_sec, failures, per_problem}`. To inspect what the wrapper actually produced (e.g. to recognize the v6A "English prose instead of Python" failure mode), read `result['failures'][i]['completion_head']` (first ~200 chars of the generation) and `result['failures'][i]['reason']` (e.g. `IndentationError: unexpected indent (<string>, line 8)` is the canonical signature of prose-as-Python). `per_problem` only carries `{task_id, passed, reason}`. Don't waste time looking for full generation arrays — they aren't logged.
- **[correction:project]** Once a v6 variant shows the v6A failure pattern at ckpt-100 LCB-10 (wrapper 0/0/0 at all T values, generations are English prose with `IndentationError` from `failures[].completion_head`), KILL the training immediately on the pod — do not wait for ckpt-200 to "confirm". The damage to the code prior is monotonic; v6L's ckpt-200 LCB-10 came back identical 0/0/0, wasting 100 more steps of pod time. User correction 2026-04-30: "why we don't kill now v6l?" The 0/0/0+IndentationError signature alone is sufficient evidence — pivot to the next variant immediately.

- **[pattern:project]** Watchdog watchdog-pgrep gotcha: when extracting the active wandb run name on the pod, do NOT just `pgrep -af finetune_phase1 | head -1` because that line is hundreds of chars and pollutes the report. Use `pgrep -af finetune_phase1 | head -1 | awk '{for(i=1;i<=NF;i++) if($i=="--wandb-run-name") print $1, $(i+1)}'` to print just `<PID> <run_name>`. Also use `LOG=$(ls -t /workspace/mythic-rdt/eval_results/V6*_RUN.log | head -1)` so the watchdog auto-detects the freshest run log without hardcoding `V6M_RUN.log` (which goes stale the moment the user pivots to v6N).

- **[correction:project]** Cross-vocab on-policy distill: forwarding the student on a freshly RIGHT-padded re-pack while sampling came from a LEFT-padded tensor silently corrupts training because RoPE puts the same logical token at a different absolute position in the two forwards — the KL trains the student against teacher logits at positions the student will never be at during inference. Always forward the student on the SAME left-padded `full_ids` used for sampling (mirror `experiments/validation/06_train_onpolicy.py` M5 structure); the teacher gets its own re-tokenized tensor since the vocabs differ. The buggy M6 lost 14.6 pp HE vs the fixed M6b (38.4 → 53.0).
- **[gotcha:project]** When per-batch elements get dropped via `continue` during cross-vocab alignment build (empty row, empty teacher tokens, `prompt_len >= seq_len`), do NOT re-use the filtered-list index `i` to look up `batch["prompt_texts"][i]` — that reads the WRONG prompt's text. Track an `orig_idx[]` of original batch positions and index `batch['prompt_texts'][orig_idx[i]]`. Same trap applies to `attention_mask[orig_idx[i]]` for computing `pad_count`.
- **[gotcha:project]** Cross-vocab `VocabMapper(multi_token="distribute")` on a 5× vocab gap (Qwen 152K → DS-Coder 32K, ~81% multi-token mass) trains the student to favour fragments over coherent identifiers — eval failure signature is NameError on undefined symbols + IndentationError + TypeError "int not callable". Use `multi_token="first_token"` instead (puts teacher mass on a single coherent next-token target). Recovers ~14 pp HE.
