# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern:project]** Long pod-side rsync: launch with `nohup rsync ... > /tmp/log 2>&1 &` and watch via `until ! kill -0 $PID; do sleep 60; done` in a `Bash run_in_background:true` task — survives Claude turn boundaries.
- **[gotcha:project]** `humaneval_smoke.py`'s `_run_base_eval_incremental` resume-from-JSON path expects each prior result to carry a `"passed"` key, but partial JSONs from a crashed prior run can lack it → `KeyError: 'passed'` at `state.results[...][r['passed']]`. When an HE-164 run crashed mid-eval and you re-launch with the same `--output-json`, the cleanest recovery is to delete the stale JSON entirely (`rm v6X_he164.json`) — partial-T deletion via JSON surgery is brittle.
- **[gotcha:project]** Pod-side `bash -u` (set -uo pipefail) launchers crash at line 1 with `WANDB_API_KEY: unbound variable` if env vars aren't propagated. `nohup bash script.sh &` and `setsid bash script.sh &` do NOT inherit interactive shell exports from `/root/.bashrc`. Read keys explicitly: `HF=$(grep '^export HF_TOKEN=' /root/.bashrc | cut -d= -f2); WB=$(grep '^export WANDB_API_KEY=' /root/.bashrc | cut -d= -f2); setsid env HF_TOKEN=$HF HUGGING_FACE_HUB_TOKEN=$HF WANDB_API_KEY=$WB bash script.sh &`.
- **[pattern:project]** Local-first sanity test for any new RecurrenceCell behavior: import `mythic_rdt.recurrence.RecurrenceCell` directly, instantiate with `hidden_size=32, n_iters=4`, feed random tensors, compare two configs (e.g. `lti_residual_scale=0.0` vs `0.01`) and assert `(out2.h_next - out1.h_next).norm() > 1e-6`. Catches plumbing bugs in <10 s before a 10 h pod run.
- **[gotcha:project]** `humaneval_smoke.py` writes `output_json` ONCE at end-of-run per T-value (atomic JSON dump, no incremental writes); intermediate state is NOT in any sqlite. Spot-checking pass@1 mid-run requires counting log lines (`grep -cE 'smoke wT=N HE.*PASS'` vs total). Per-T progress in the log uses prefix `[smoke wT=<T> HE]` or `[smoke wT=<T> LCB]`; BASE phase prefix is just `[smoke]`.
- **[gotcha:project]** `humaneval_smoke.py`'s `_run_base_eval_incremental` resume-from-JSON path expects each prior result to carry a `"passed"` key — partial JSONs from a crashed prior run can lack it → `KeyError: 'passed'`. When an HE-164 run crashed mid-eval and you re-launch with the same `--output-json`, the cleanest recovery is to delete the stale JSON entirely (`rm v6X_he164.json`) — partial-T deletion via JSON surgery is brittle.
- **[gotcha:project]** `humaneval_smoke.py`'s smoke loader injects DepthLoRA from CLI defaults (`self_attn.q_proj_or_q_a, self_attn.o_proj` rank 8) and then calls `_load_trainable_state` — when the ckpt was trained with a different LoRA tree (e.g. FFN-on-experts rank 24), the load silently no-ops with `loaded 4 trainable tensors  missing=76 unexpected=7410` and the wrapper runs at INIT, making T=1=T=2=T=4=BASE. ALWAYS verify post-load report shows `missing=0 unexpected=0`. The smoke now autodetects targets+rank from the saved state_dict by default; pass `--no-lora-autodetect` only for intentional partial-load ablations.
- **[gotcha:project]** `margin_alpha > 0` in `src/mythic_rdt/training/trainer.py` triggers dual-T mode that BYPASSES the curriculum schedule — `T_hi` (e.g. 4) is sampled every step regardless of `--curriculum-warmup-steps`/`--curriculum-phase2-start`. Useful invariant for dual-T runs, but means schedule flags are inert when margin loss is active; don't tune them expecting effect.
- **[gotcha:project]** LTI in this codebase is contractive by construction — `A_diag` is parameterized as `Diag(-exp(log_A))` so spectral radius < 1 is structural. Do NOT add a clamp to `lti_residual_scale` analogous to the LayerScale clamp; the LTI cell cannot blow up ||h|| even unbounded.
- **[pattern:project]** When adding a new `RecurrenceCell` hyperparameter, plumb it through ALL FOUR sites or it silently defaults: (1) `configuration.py` config kwarg + self-assign, (2) `modeling.py` RecurrenceCell construction with `getattr(config, ..., default)`, (3) `recurrence.py` `__init__` kwarg + self-assign + forward use, (4) CLI in `scripts/06_train_sft_recurrence.py` + `scripts/finetune_phase1.py` + `scripts/humaneval_smoke.py`. Missing the smoke script means eval re-loads ckpts at the default value, masking a real training effect.
- **[pattern:project]** When `claude-consultants` HTTP endpoints return 404 for a sid (`session not found`), fetch the persisted output from disk instead: `.claude-hooks/consultants/<sid>/summary.md` (and `transcript.md`). Council sessions are written to disk by the hook even when the in-memory server has rotated them.
- **[convention:project]** Decision rule for any v6 wrapper variant: T=4 ≥ target pp AND monotonicity T=4 > T=2 > T=1 AND HE-164 ≥ base 70.7% (no code-prior tradeoff). LCB-medium-full n=55 is ~1.2 SE / problem — a +4 problem gain is necessary not sufficient; require all three legs before declaring a v6X passes.
- **[pattern:project]** Pod-side training progress can be extracted in one ssh call: `pgrep -af finetune_phase1 | grep -v 'bash -c' | head -1` (PID) + `grep -E 'ce_gap|margin_loss|teacher_refine|Traceback|exit_code|ALL DONE' eval_results/<RUN>.log | tail -3` (loss snapshot) + `grep -oE '[0-9]+/[0-9]+ \[[^]]+\]' | tail -1` (step+ETA) + `ls checkpoints/<dir>/ | grep checkpoint` (fresh ckpts) + `nvidia-smi --query-gpu=utilization.gpu,memory.used --format=csv,noheader`. tqdm writes `\r`-progress so plain `tail` shows the live line; no need to convert CRs.
- **[gotcha:project]** Inside `ssh '...'` single-quoted commands, `awk` field refs must be escaped as `\$1` (the outer single-quotes already protect from local shell expansion, but `awk "{print $1}"` inside double-quoted awk script needs `\$1` to defer expansion to the remote shell). Mis-escape = silent empty PID output.
- **[pattern:project]** Recurring health-check crons should fire on prime-ish minutes (`7,27,47 * * * *` for 20-min cadence; `46 * * * *` for hourly) to avoid the global :00/:30 stampede that hits the API simultaneously. Already-running crons can be retuned via `CronDelete` + `CronCreate`; user may request cadence changes mid-run ("change from 20 min to 1 hour") — delete the high-frequency job, keep the hourly.
- **[pattern:project]** When an eval script crashes mid-multi-phase (HE Phase 2a fails but LCB Phase 2b is still running) and you want to re-run just the failed phase WITHOUT killing the in-flight one: write a waiter script (`while pgrep -f 'humaneval_smoke.py.*<phase_marker>'; do sleep 30; done`) and `nohup` it in the background. It auto-launches the rerun the moment the GPU frees.
- **[gotcha:project]** In `recurrence.py`'s block_mode_residual branch, the LTI `injection` was historically computed but DROPPED from `h_next` (only `h + mix * (block_out - h)` was used), making `LTIInjection.A_diag` and `B_proj` dead weight (no forward use → no gradient → no learning). Any v6 variant relying on LTI signal must verify `lti_residual_scale > 0.0` is plumbed through `MythicRDTDeepseekV2Config` → `RecurrenceCell` → forward.
- **[gotcha:project]** `humaneval_smoke.py` writes `output_json` ONCE at end-of-run with FULL multi-phase aggregate (`results: [base_he, wrapper_T1_he, wrapper_T2_he, wrapper_T4_he, (+ lcb variants)]`, each with `per_problem: [{task_id, passed, reason}, ...]`). Mid-run, the file is the BASE incremental snapshot only (looks like single-phase data because it IS, temporarily) — `_run_base_eval_incremental` writes atomic temp+rename per batch for resume-after-kill. DO NOT conclude "overwritten per phase" or propose restart from mid-run state. Decision rule: if `pgrep humaneval_smoke` finds a PID OR log lacks `[smoke] wrote -> ...`, the JSON is incomplete; wait for that line then inspect. For byte-level base-vs-wrapper diff use `results[i].per_problem`.
- **[pattern:project]** Spot-check pass@1 by grepping the run log: `grep -cE "smoke wT=$T HE" $LOG` for total + `... HE.*PASS` for passes. Pair with `grep -E 'PHASE|exit_code|wrote ->' $LOG` to confirm phase boundaries. Per-T progress prefix is `[smoke wT=<T> HE]` / `[smoke wT=<T> LCB]`; BASE phase prefix is `[smoke]`.
- **[gotcha:project]** Identical pass@1 across BASE/T=1/T=2 (e.g. all `70.7% 116/164`) is NOT evidence the wrapper degenerated to base — it can mean exactly the same problems passed, with very different generations. Verify wrapper is active by mean `gen_secs` per task: T=2 should be ~2x T=1, T=4 ~4x (e.g. wT=1 mean=12.8s, wT=2 mean=24.8s, wT=4 mean~50s+). Sample-set identity ≠ token identity; the wrapper-positive-direction test is LCB pass@1 delta + which-tasks-flipped, not the aggregate number.
