# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern:project]** Sequential GPU-job handoff via single Monitor: `until grep -q 'DONE' <log>; do sleep 30; done; sleep 15; nvidia-smi ...; nohup ... &`. The `sleep 15` after DONE lets CUDA fully release VRAM before the next job's `from_pretrained` allocator races for the same memory.
- **[gotcha:project]** Monitor tool bodies run in a bare shell that has NOT sourced `/root/anaconda3/etc/profile.d/conda.sh` — wrapping `conda run -n <env> python ...` fails with `nohup: failed to run command 'conda': No such file or directory`. Fix: source the conda init script first inside the Monitor body, or call the env's python directly: `/root/anaconda3/envs/mythic-rdt/bin/python -u`.
- **[pattern:project]** When chaining `ckpt-N lands → wait for GPU → fire eval` triggers concurrently, gate each launch on a `pgrep -f 'python.*humaneval_smoke'` busy-loop with `sleep 60; nvidia-smi --query-gpu=memory.free` BEFORE the eval kicks off. Without it, two checkpoints landing close together race for the 12 GB free window on the 3090.
- **[env:project]** Pandorum (hostname `wsl`, RTX 5080 16GB Blackwell sm_120) is the dev-time spillover compute when the main 3090 is busy and the pod is mid-training. Conda at `/home/claude_test/miniconda3`, project mounted at `/mnt/backup_models/Mythic-RDT` via CIFS, env `mythic-rdt`: torch 2.10.0+cu128, transformers 4.46.3, bitsandbytes 0.49.2, flash-attn NOT installed.
- **[correction:project]** For base-model eval suites used as wrapper comparison baselines, run ONLY LCB-medium full — skip LCB-easy and LCB-hard. User directive 2026-04-30. Authoritative pandorum 5080 base baselines (DS-Coder-V2-Lite NF4): HE-164=70.1%, LCB-medium-55=21.8%, MBPP+-378-extended=67.7%, MBPP-basic=80.4%.
- **[gotcha:project]** Self-deadlocking trigger Monitor: a Monitor body that uses `pgrep -f 'humaneval_smoke|local_v6k_ckpt_lcb10'` to wait for "another eval to finish" will match ITS OWN bash command line and loop forever. Fix: narrow the pgrep to the actual python process (`pgrep -f 'python.*humaneval_smoke'`).
- **[gotcha:project]** Cached NF4 base (`base/DeepSeek-Coder-V2-Lite-Instruct-nf4/`) loads ONLY in `mythic-rdt` conda env (transformers 4.46.3). The `lightseek/.venv` (transformers 5.5.0) crashes ~30s into the load with `NotImplementedError: "normal_kernel_cuda" not implemented for 'Byte'` because tf 5.x re-runs init on every weight including uint8 NF4 quant_state tensors. Bug-073.
- **[gotcha:project]** When cloning a `local_v6X_ckpt_lcb10.sh` script for a new variant, copy the CONDA ACTIVATION BLOCK verbatim from a recent working sibling — don't write it fresh. The trap is reflexively typing `source /shared/dev/lightseek/.venv/bin/activate` when the working version uses `source /root/anaconda3/etc/profile.d/conda.sh && conda activate mythic-rdt`.
- **[gotcha:project]** Eval JSON result records (`eval_results/v6X_ckptN_lcb10_local.json`) do NOT contain the actual generation strings under `generations` or `samples` — those return 0-length lists. Per-result schema is `{name, pass_at_1, n_pass, n_total, elapsed_sec, failures, per_problem}`. Read `result['failures'][i]['completion_head']` (first ~200 chars) and `result['failures'][i]['reason']` (e.g. `IndentationError: unexpected indent` is the canonical signature of prose-as-Python).
- **[correction:project]** Once a v6 variant shows the v6A failure pattern at ckpt-100 LCB-10 (wrapper 0/0/0 at all T values, generations are English prose with `IndentationError`), KILL training immediately on the pod — do not wait for ckpt-200 to "confirm". - **[pattern:project]** Pod training launch with credentials inline (vast.ai): `ssh -p 22024 -o StrictHostKeyChecking=no root@ssh6.vast.ai "cd /workspace/mythic-rdt && WANDB_API_KEY=<key> HF_TOKEN=<key> nohup bash scripts/pod_runner/run_v6X.sh > /workspace/mythic-rdt/eval_results/V6X_LAUNCH.log 2>&1 & echo PID=\$!"`.
- **[pattern:project]** Loss-line tail extraction from wandb-logged training: use `tail -40 $LOG | grep -oE '(ce_gap|kl_anchor|margin_loss).{0,18}'` — fixed-length context after each key works around the dict format. Avoid `grep -oE "'(ce_gap|...)':\s*[-0-9.e]+"` because the quote-escape inside an `ssh "..."` outer command breaks the regex.
- **[correction:project]** Cross-vocab on-policy distill: forwarding the student on a freshly RIGHT-padded re-pack while sampling came from a LEFT-padded tensor silently corrupts training because RoPE puts the same logical token at a different absolute position in the two forwards. Always forward the student on the SAME left-padded `full_ids` used for sampling.
- **[gotcha:project]** When per-batch elements get dropped via `continue` during cross-vocab alignment build, do NOT re-use the filtered-list index `i` to look up `batch["prompt_texts"][i]` — that reads the WRONG prompt. Track an `orig_idx[]` of original batch positions and index `batch['prompt_texts'][orig_idx[i]]`.
- **[gotcha:project]** Cross-vocab `VocabMapper(multi_token="distribute")` on a 5× vocab gap (Qwen 152K → DS-Coder 32K, ~81% multi-token mass) trains the student to favour fragments over coherent identifiers — failure signature is NameError + IndentationError + TypeError. Use `multi_token="first_token"` instead. Recovers ~14 pp HE.
- **[gotcha:project]** SQLite `sqlite3 db.db "SELECT ... WHERE task=\"humaneval\""` over `ssh "..."` double-escapes and parses `humaneval` as a column name ("no such column: humaneval"). Fix: use single quotes inside the SQL (`task='humaneval'`) AND wrap the outer ssh in double-quotes, OR write the SQL to a heredoc file (`cat > /tmp/q.sql <<'EOF' ... EOF`) and pipe `sqlite3 db.db < /tmp/q.sql`. The heredoc form with quoted `'EOF'` prevents shell-side variable expansion.
- **[gotcha:project]** Eval `--chat-template` flag wraps prompts via `tokenizer.apply_chat_template` with `add_generation_prompt=True`. DSC-V2-Lite under chat template emits prose preambles ("To solve this problem...") that the raw `exec(prompt+gen)` scorer cannot execute, dropping HE pass@1 from 75.6% (raw) → 10.4% (chat). For BASE re-eval baselines that need to be apples-to-apples with raw-completion training recipes (M37c et al), do NOT pass `--chat-template`; or use a chat-aware scorer that strips fenced code blocks before exec.
- **[gotcha:project]** Caliber stop-hook `caliber-check-sync.sh` blocks every Claude Code `Stop` event in a git repo without `caliber` in `.git/hooks/pre-commit` — manifests as nag prompt suggesting `/setup-caliber` even though both /shared/dev/cross-tokenizer-distill and /srv/.../Mythic-RDT have CALIBER_LEARNINGS.md. Run `/setup-caliber` once per repo to install the hook, or accept the nag.
- **[gotcha:project]** Eval JSON result records (`eval_results/v6X_ckptN_lcb10_local.json`) do NOT contain the actual generation strings under `generations` or `samples` — those keys return 0-length lists. Per-result schema is `{name, pass_at_1, n_pass, n_total, elapsed_sec, failures, per_problem}`. Read `result['failures'][i]['completion_head']` (first ~200 chars) and `result['failures'][i]['reason']` (e.g. `IndentationError: unexpected indent` is the canonical signature of prose-as-Python). `per_problem` only carries `{task_id, passed, reason}`.

- **[pattern:project]** Loss-line tail extraction from wandb-logged training: training emits one-line dicts like `{'ce_lo_T': 2.59, 'ce_hi_T': 2.59, 'ce_gap': 0.0, ...}` interleaved with tqdm progress bars. Use `tail -40 $LOG | grep -oE '(ce_gap|kl_anchor|margin_loss).{0,18}'` — fixed-length context after each key works around the dict format. Avoid `grep -oE "'(ce_gap|...)':\s*[-0-9.e]+"` because the quote-escape inside an `ssh "..."` outer command breaks the regex.
- **[correction:project]** Cross-vocab on-policy distill: forwarding the student on a freshly RIGHT-padded re-pack while sampling came from a LEFT-padded tensor silently corrupts training because RoPE puts the same logical token at a different absolute position in the two forwards. Always forward the student on the SAME left-padded `full_ids` used for sampling; the teacher gets its own re-tokenized tensor since the vocabs differ.
- **[gotcha:project]** When per-batch elements get dropped via `continue` during cross-vocab alignment build (empty row, empty teacher tokens, `prompt_len >= seq_len`), do NOT re-use the filtered-list index `i` to look up `batch["prompt_texts"][i]` — that reads the WRONG prompt. Track an `orig_idx[]` of original batch positions and index `batch['prompt_texts'][orig_idx[i]]`. Same trap applies to `attention_mask[orig_idx[i]]` for `pad_count`.
- **[gotcha:project]** Cross-vocab `VocabMapper(multi_token="distribute")` on a 5× vocab gap (Qwen 152K → DS-Coder 32K, ~81% multi-token mass) trains the student to favour fragments over coherent identifiers — failure signature is NameError + IndentationError + TypeError "int not callable". Use `multi_token="first_token"` instead. Recovers ~14 pp HE.
- **[pattern:project]** Cache the NF4-quantized DS-Coder-V2-Lite-Instruct base ONCE via `scripts/cache_nf4_base.py`. Saves with `BitsAndBytesConfig(load_in_4bit=True, bnb_4bit_quant_type='nf4', bnb_4bit_compute_dtype=bf16, bnb_4bit_use_double_quant=True)` + copies tokenizer + all `*.py *.json *.md` files (required so `trust_remote_code=True` works). Subsequent loads use `--quant none`. Drops load wall from 10-15 min to 30-60s.
- **[gotcha:project]** SQLite via `ssh "sqlite3 db ..."` double-quote escaping: `WHERE task=\"humaneval\"` is parsed as a column reference ("no such column: humaneval"). Two reliable forms: (a) inner SQL uses single quotes for string literals (`task='humaneval'`) inside a double-quoted ssh command; (b) write the SQL to a heredoc file with quoted `'EOF'` (`cat > /tmp/q.sql <<'EOF' ... EOF`) and pipe `sqlite3 db < /tmp/q.sql` — heredoc form prevents shell variable + quote expansion entirely.
- **[gotcha:project]** Eval `--chat-template` flag in `06_eval_batched.py` wraps prompts via `tokenizer.apply_chat_template(..., add_generation_prompt=True)`. DSC-V2-Lite under chat template emits prose preambles ("To solve this problem...", "Here's the implementation...") that the raw `exec(prompt+gen)` scorer cannot execute, dropping HE pass@1 from 75.6% (raw) → 10.4% (chat). Either (a) skip `--chat-template` for raw-completion baselines, or (b) ensure the SFT trainer used `--chat-template` so train and eval prompt formats match — the `06_train_sft_on_teacher.py` script gained a `--chat-template`/`--system-prompt` flag pair for exactly this reason.
- **[gotcha:project]** OOM at `batch_size=16` on 44 GB GPU with FA2 + DSC-V2-Lite + chat-template + 1024 max-new-tokens: drop to `batch_size=8`. After kill, prune the partially-cached rows from the SQLite cache db (`DELETE FROM results WHERE ...`) before relaunch so the resume regenerates them at the smaller batch.
- **[gotcha:project]** Caliber stop-hook `caliber-check-sync.sh` blocks every Claude Code `Stop` event in a git repo without `caliber` in `.git/hooks/pre-commit`. Run `/setup-caliber` once per repo to install the hook (Mythic-RDT is wired; cross-tokenizer-distill needed setup as of 2026-05-07).
- **[convention:project]** Pin BASE/M-series eval JSONs (`baselines/<model>/*.json`) to the Mythic-RDT repo as soon as they're generated. Vast.ai pod destruction wipes `/workspace/` results — only what's git-committed survives. Add a short `baselines/<model>/README.md` that records the exact eval flags, the stack (FA2 / chat-template / merge_and_unload), and reconciles against any prior measurements when re-evaluation drifts (e.g. 75.6/60.6 from a destroyed 4090 pod vs 73.8/64.6 on a fresh 6000 Ada — both exist in history; the canonical baseline is the one with a committed JSON).
