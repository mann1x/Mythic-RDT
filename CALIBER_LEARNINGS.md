- **[gotcha:project]** When threading KV cache through the wrapper, `_prepare_4d_causal_attention_mask(attention_mask, (bsz, seq_len), inputs_embeds, past_key_values_length=past_kv_len)` MUST be called with the correct `past_kv_len` (read via `past_key_values[0].get_seq_length(0)`). Otherwise DS-V2's attention raises `ValueError: Attention mask should be of size (bsz, 1, q_len, kv_seq_len), but is torch.Size([bsz, 1, 1, ...])` on incremental decode.
- **[gotcha:project]** Long-running ssh commands (e.g., `pkill -KILL ...; sleep 2; pgrep -af ...; nvidia-smi`) can hang silently and return empty responses — Bash tool reports success but `stdout=""`. Add `-o ConnectTimeout=15` to the ssh invocation to surface the actual issue; on retry the same command typically succeeds.
- **[gotcha:project]** Wrapper KV cache + RoPE bug (bug-049): the wrapper's `forward()` originally computed `position_ids = arange(past_kv_len, past_kv_len + seq_len)` uniformly across the batch, ignoring that left-padded sequences have content starting at different offsets. Base.generate handles this via `prepare_inputs_for_generation` which does `cumsum(attention_mask) - 1`. With `padding_side='left'` + KV cache, RoPE encoded content tokens at absolute-buffer positions (shifted by pad_len) on the prefix pass, then incremental tokens got position `[past_kv_len]` — cached K/V written at the wrong RoPE phase, never overwritten. Fix: when `attention_mask` is 2D, use `position_ids = attention_mask.long().cumsum(-1) - 1; masked_fill_(attention_mask==0, 1)`, then slice trailing `seq_len` for incremental decode. HE-easy short prompts (~80 tokens, similar lengths) masked the bug at 95%; LCB-medium (~300+ tokens, high length variance) hit 0%.
- **[gotcha:project]** Stale `.pyc` shadowing on the pod: after `scp`'ing edited `src/mythic_rdt/*.py` to the pod, the existing `.pyc` in `src/mythic_rdt/__pycache__/` can shadow the patched `.py` even when timestamps look fresh. Empirically (bug-049): same patched `modeling.py` gave wrapper T=1 LCB-medium 0% with stale pyc, then 30% (matching base) once `__pycache__` was purged. ALWAYS prefix any pod-side launch that imports a patched module with `rm -rf /workspace/mythic-rdt/src/mythic_rdt/__pycache__`. Same general gotcha as the HF dynamic-module cache for `base/<MODEL>/modeling_*.py`, but for in-tree wrapper code.

- **[pattern:project]** When wiring v5-style dual-T training, the trainer's `compute_loss` activates dual-T iff `margin_alpha > 0` OR `distill_alpha > 0` (otherwise single-T curriculum path). The dual-T branch does TWO `model(...)` calls per microbatch (at `t_lo` and `t_hi`) — wall-clock per step roughly doubles vs the v4 single-T path. Plan ETAs accordingly: v5 probe @ 1×3090 NF4 ≈ 120 s/step → 200 steps ≈ 7 h. Logged scalars: `margin_loss`, `distill_loss`, `ce_lo_T`, `ce_hi_T`, `ce_gap`, `T_lo`, `T_hi`, `kl_anchor` (when also enabled). Watch `ce_gap` trend — positive at init means high-T is currently worse and the margin term is doing its job.
