# Caliber Learnings

Accumulated patterns and anti-patterns from development sessions.
Auto-managed by [caliber](https://github.com/caliber-ai-org/ai-setup) — do not edit manually.

- **[pattern:project]** When recurrence training shows pass@1 collapse at T>1, before tweaking loss weights or LayerScale clamps, run a per-iteration **hidden-state drift probe** that captures `||h_t||`, `||h_t - h_0||`, `cos(h_t, h_0)`, and per-step `||h_t - h_{t-1}||` at every iteration t∈[0,T], plus T=1-vs-T=T argmax-flip distribution across early/mid/late thirds of the sequence. Three competing hypotheses (norm growth / lm-head fragility / cascade) discriminate cleanly. Implementation: monkey-patch `wrapper._loop_step` to append `h.detach().to(torch.float32).cpu()` per iteration; ~3s/prompt at T=4 with 4-bit base on a 3090. Reference: `scripts/_probe_recurrence_drift.py`.
- **[pattern:project]** Parallelize independent A/B test variants across pod + local 3090 with an env-var variant filter: `MYTHIC_FIXTEST_VARIANTS='0,1,2'` runs only those indices and writes to `*_subset_0-1-2.json` (avoid output collision). Halves wall-clock when GPU is the bottleneck. Apply when each variant is independent (e.g. one-time inference probe across N anchor recipes), not when state must be shared.
- **[pattern:project]** When invoking `wrapper_generate(...)` from the Mythic-RDT smoke harness, the signature requires `device` as a positional/keyword argument — `device=torch.device(device)` — not just `batch_size`. Forgetting it produces `TypeError: wrapper_generate() missing 1 required positional argument: 'device'` at the first call site, killing the test before any output is written. Reference signature lives in `scripts/humaneval_smoke.py:567` — re-read it before authoring any new probe that calls it.
- **[gotcha:project]** `nohup conda run --no-capture-output -n <env> python ...` fails with `nohup: failed to run command 'conda': No such file or directory` because nohup's PATH does not include `/root/anaconda3/condabin`. For backgrounded conda-env scripts, ALWAYS use the env's interpreter directly: `nohup /root/anaconda3/envs/<env>/bin/python -u <script> > <log> 2>&1 & disown`. `conda run` works inside an interactive shell but not under nohup. Applies to ANY non-shell command wrapper (nohup, setsid, env, sudo -E without `bash -c`).
- **[gotcha:project]** When SSHing a command that references `$WANDB_API_KEY` in the remote shell, it expands on the REMOTE pod (which has no such env var by default) — NOT on the local solidPC. The remote training process gets `WANDB_API_KEY=` (empty) and disables wandb silently with `WANDB_API_KEY not in env; wandb disabled`. Always inline the literal value: `ssh <pod> "... WANDB_API_KEY=503a370c... nohup bash <script>"` — or pre-extract via `KEY=$(grep -A2 api.wandb.ai /root/.netrc | awk '/password/{print $2}'); ssh <pod> "... WANDB_API_KEY=$KEY ..."` so the local shell expands `$KEY` BEFORE SSH transport. Verify after launch with `cat /proc/<pid>/environ | tr '\0' '\n' | grep WANDB_API_KEY`.
- **[gotcha:project]** Vast.ai pod GPU memory may show another tenant's process holding GB of VRAM in `nvidia-smi --query-compute-apps`. RTX 6000 Ada is 48GB total but a co-tenant `Process 1264918 has 35.70 GiB memory in use` leaves only ~10GB free — your training OOMs at step 7. Always `nvidia-smi --query-gpu=memory.free --format=csv,noheader` BEFORE launching, and `nvidia-smi --query-compute-apps=pid,used_memory,process_name --format=csv` to confirm only your own PIDs are present. If a foreign PID shows up, the pod is shared — destroy and re-rent or accept the smaller effective budget.
- **[gotcha:project]** When relaunching pod training after a `pkill -KILL`, the previous CUDA process can leave VRAM allocated for several seconds AFTER the PID is gone. The new launch races the cleanup and OOMs at step 7 with `Process <old_pid> has 35.70 GiB memory in use` even though `pgrep` shows clear. Always wait for `nvidia-smi --query-gpu=memory.used --format=csv,noheader` to read `0 MiB` (or `1 MiB`) before relaunching, not just for `pgrep` to be empty.
- **[pattern:project]** When adding a new architectural variant flag to the Mythic-RDT wrapper (e.g. `block_mode_residual`), it MUST be plumbed through THREE call sites: (1) `MythicRDTDeepseekV2Config(...)` field in `src/mythic_rdt/configuration.py`, (2) `argparse` + `cfg = MythicRDTDeepseekV2Config(..., <flag>=args.<flag>, ...)` in `scripts/finetune_phase1.py`, (3) same argparse + cfg construction in `scripts/humaneval_smoke.py`. Forgetting (3) silently loads the checkpoint with the WRONG architectural formula at eval time. Always grep `grep -n '<existing-flag>' scripts/finetune_phase1.py scripts/humaneval_smoke.py src/mythic_rdt/configuration.py` to find all call sites before adding a new flag.
- **[pattern:project]** Before launching a long architectural-fix training run, sanity-check the formula's behavior in a 30-line CPU-only smoke test: instantiate the cell with toy hidden_size=2048 + 10 tokens, simulate a synthetic `block_out = h * factor` with the empirically-observed factor, run T iterations, print `||h||` per iteration. - **[pattern:project]** Trivial-solution trap diagnostic for recurrence training: when `ce_gap = wrapper@T_hi_CE - wrapper@T_lo_CE = 0.0` for many consecutive steps AND `kl_anchor` stays in 1e-5 noise floor, training has NO gradient signal — wrapper@T_hi == wrapper@T_lo == base byte-for-byte. Root cause is usually `first_iter_identity=True` combined with too-small `layerscale_init` (e.g. 1e-4 → mix=5e-5): t=0 bypasses the cell, t=1..3 perturbations are too tiny to differentiate outputs. KILL the run before step 100 and raise `layerscale_init` (e.g. to 0.05 → mix=0.025) so gradient signal exists. Don't merely drop `first_iter_identity` without raising LS — that breaks T=1 base parity (coda receives ||~25|| prelude output instead of ||~85|| block output → catastrophic OOD ce_lo~18 nats from step 0).
- **[pattern:project]** Set up 30-min cron health checks for long pod training runs with explicit KILL conditions encoded in the prompt itself: K1=trivial-solution trap (kl_anchor noise + ce_gap=0 for 20 steps post-warmup), K2=base parity broken (ce_lo_T diverges >2 nats from base for 10+ steps), K3=catastrophic divergence (ce_gap>1 growing for 10+ steps), K4=process dead (no exit_code=0 in tail), K5=GPU<20% util for 10 min. Use `CronCreate` with `recurring: true` at `"17,47 * * * *"` (off-the-hour minutes to avoid fleet-wide alignment). Each fire SSHes the pod, extracts metrics, applies the kill rubric autonomously.
- **[correction:project]** NEVER auto-kill v6 training runs based on cron health-check K1-K5 conditions. Investigate, run interim LCB eval if checkpoint is fresh, present findings + comparison vs prior v6 attempts (v6A T=4=0%, v6E T=4=0%, v6H ckpt-100 T=4=20%), and ASK the user before killing. What looks "degraded" at ckpt-100 may still be the best v6 result we've ever achieved relative to prior failures. Cron prompts must say "INVESTIGATE+ACTION+ASK" not "KILL+REASON". User directive 2026-04-29 after v6H ckpt-100 results triggered premature-kill instinct.
- **[correction:project]** Do NOT trust the `first_iter_identity=True` invariant past INIT (step 0). The invariant only makes the recurrence ADD zero at t=0 (`h_next = block_out`); it does NOT make the BLOCK forward equal to base — recurrent block layers carry DepthLoRA on q_proj/o_proj that drifts away from zero during training. After ANY training step that updates LoRA-B[0], T=1 ≠ base byte-exact. Whenever evaluating a TRAINED checkpoint at T=1, always re-run base on the SAME `--lcb-limit/--lcb-difficulty/--lcb-min-date` problem set in the same session (selection is deterministic by these flags). We've been bitten twice assuming T=1=base by invariant.
- **[gotcha:project]** Local 3090 OOM during Mythic-RDT eval is often caused by ollama models squatting on the GPU (e.g. `gemma4:e2b` 8GB + `qwen3-embedding:0.6b` 7GB = 15GB stolen, leaving ~9GB free → OOM at LayerNorm softmax allocation). DIAGNOSE with `nvidia-smi --query-compute-apps=pid,used_memory,process_name --format=csv,noheader` (look for `/usr/local/bin/ollama` PIDs) and `curl -s http://localhost:11434/api/ps` (lists loaded models with size_vram and expires_at). UNLOAD gracefully via `curl -s -X POST http://localhost:11434/api/generate -d '{"model":"<m>","keep_alive":0}'` for each loaded model — NEVER `kill` the ollama PID, that can leave VRAM allocated for seconds.
- **[pattern:project]** When SCPing a Mythic-RDT checkpoint from pod to local for inference-only eval, fetch ONLY the four inference essentials: `mythic_rdt_trainable.pt`, `mythic_rdt_config.json`, `mythic_rdt_manifest.json`, `mythic_rdt_curriculum.json`. Skip `optimizer.pt`, `rng_state.pth`, `scheduler.pt`, `trainer_state.json`, `training_args.bin` — those are ~22MB extra and only needed for training resume. Reduces transfer to ~20MB and avoids accidentally loading huge trainer_state.json into memory at smoke launch.
- **[gotcha:project]** `scripts/humaneval_smoke.py` auto-resolves `--checkpoint <parent_run_dir>` to the LATEST `checkpoint-N` subdir via `sorted(ckpt_dir.glob("checkpoint-*"), key=lambda p: int(p.name.split("-")[1]))[-1]` (smoke.py:923). To eval a SPECIFIC older checkpoint, pass the full subdir path `--checkpoint <run_dir>/checkpoint-200` — passing the parent picks the highest-numbered ckpt regardless of which one you intended. Verify in log via `[smoke] resolved latest sub-checkpoint: checkpoint-N` line.
- **[pattern:project]** Per-problem failure attribution for LCB eval: parse the `failures` array in each result, build `base_fail_ids = set(f['task_id'] for r in base_results for f in r.get('failures',[]))`, then for each wrapper T compute `wrap_fails - base_fails = NEW BREAKS` (problems base passes but wrapper fails) and `base_fails ∩ wrap_passes = RECOVERIES`. Tracks two distinct training signals — "is recurrence harmful?" (new breaks count) vs "is recurrence productive?" (recoveries count). Note: `failures` array is capped (~5-8 stored), so use `n_total - n_pass` for the true failure count and treat per-task analysis as approximate when capped.
- **[gotcha:project]** LCB problem selection in `humaneval_smoke.py` is deterministic by `(--lcb-limit, --lcb-difficulty, --lcb-min-date)` — same flags = same problem set, period. Use this property to compare a trained wrapper's LCB-N to the base's LCB-N exactly. Do NOT use `--lcb-limit 30` for wrapper if base ran at `--lcb-limit 10` and try to align by task_id post-hoc — just re-run with matching flags. The deterministic selection lives in `load_lcb()` at smoke.py:244 (filters jsonl shards by difficulty + contest_date).
- **[pattern:project]** When adding a new TRAINER kwarg (e.g. `focal_gamma`, distinct from architectural-wrapper flags), plumbing requires THREE sites: (1) `MythicRDTTrainer.__init__` in `src/mythic_rdt/training/trainer.py` — add kwarg + `self.<name> = ...` + diagnostic print, (2) argparse `p.add_argument("--<name>", ...)` in `scripts/finetune_phase1.py`, (3) `MythicRDTTrainer(... <name>=args.<name>, ...)` call site near `scripts/finetune_phase1.py:459`. Distinct from wrapper-config plumbing — don't conflate the two when grepping. Sanity-check after edit with `PYTHONPATH=src python -c "from mythic_rdt.training import MythicRDTTrainer; import inspect; print(inspect.signature(MythicRDTTrainer.__init__).parameters['<name>'].default)"` AND `PYTHONPATH=src python scripts/finetune_phase1.py --help | grep -A 2 -- '--<name>'`.
- **[gotcha:project]** `python -c "exec(open('scripts/<file>.py').read())"` fails with `NameError: name '__file__' is not defined` when the script references `__file__` (most scripts in this repo do, for path resolution). For sanity-checking argparse/imports outside a full CLI run, use direct module imports: `PYTHONPATH=src python -c "from mythic_rdt.training import MythicRDTTrainer; ..."` and `PYTHONPATH=src python scripts/<file>.py --help` — do NOT exec the script body via open/read.
- **[gotcha:project]** Ollama VRAM does NOT free immediately when you POST `keep_alive=0` to `/api/generate` OR run `ollama stop <model>` — both return success but the runner subprocess (`ollama runner --ollama-engine --model <blob>`) keeps the model resident until its TTL expires or another model is requested. `nvidia-smi --query-compute-apps` will still show the runner PID holding the VRAM. To truly verify GPU is free before launching an eval, poll `nvidia-smi --query-compute-apps=pid,process_name --format=csv,noheader | grep -v ollama` is empty AND `memory.used < 1000 MiB` — don't trust the API's 200 OK.
- **[pattern:project]** When fetching a Mythic-RDT checkpoint sub-directory from a pod, SCP only the four inference essentials per checkpoint: `mythic_rdt_trainable.pt`, `mythic_rdt_config.json`, `mythic_rdt_manifest.json`, `mythic_rdt_curriculum.json`. Mirror the pod path locally (`mkdir -p checkpoints/<run>/checkpoint-N` first), then `scp -P <port> -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null root@<host>:<path>/{trainable.pt,config.json,manifest.json,curriculum.json} <local>/`. Skip optimizer/scheduler/RNG/trainer-state — those are training-resume artifacts only.
- **[pattern:project]** When fetching final eval JSONs from a pod, batch the SCP — `scp -P <port> root@<host>:<dir>/v6h_he20.json root@<host>:<dir>/v6h_lcb30.json <local>/eval_results/`. Reduces ssh handshake overhead vs separate calls. After fetch, parse both with one inline python: `python3 -c "import json; r=json.load(open(p)); [print(f\"{x['name']:20s} pass@1={x['pass_at_1']*100:.1f}% ({x['n_pass']}/{x['n_total']})\") for x in r['results']]"` — gives a clean A/B table in 5s.
- **[gotcha:project]** HF Trainer's stdout interleaves tqdm carriage-return progress bars (`\r 26%|██▌ | 104/400 ...`) with the per-step metric dicts (`{'ce_lo_T': ..., 'ce_gap': ...}`). A naive `tail -n 50 <log>` returns mostly progress-bar garbage with maybe 5 metric lines. Always extract metric series via `tail -200 <log> | grep -E "ce_lo_T|margin_loss|kl_anchor" | tail -50` — grep filters out the tqdm noise so the trend window is actually 50 metric points instead of 5.
- **[pattern:project]** Before launching a new pod-side training variant (e.g. v6I after v6H), run a 5-line pre-flight: (1) `ps -ef | grep finetune_phase1 | grep -v grep` (must be empty — old run killed); (2) `nvidia-smi --query-gpu=memory.used --format=csv,noheader` (must be ≤1 MiB — wait if cleanup race); (3) `df -h /workspace | tail -1` (need ≥80GB free for ckpts); (4) `ls /workspace/mythic-rdt/checkpoints/` (verify no orphan ckpt dir from prior run); (5) `ls /workspace/mythic-rdt/scripts/pod_runner/run_v6X.sh` (the runner script must already be SCPed). Skip any of these and you discover the failure 30s into a $5 launch.
- **[gotcha:project]** When evaluating a Mythic-RDT checkpoint via `scripts/humaneval_smoke.py`, ALL architectural flags from the training launch must be re-passed at eval — `--block-mode`, `--block-mode-residual`, `--first-iter-identity`, `--gate-init-bias`, `--layerscale-clamp-max <ckpt's value>`, `--prelude-layers`, `--coda-layers`, `--recurrent-block-start`, `--recurrent-block-end`, `--max-loop-iters`. The ckpt loads only the trainable state (`mythic_rdt_trainable.pt`); everything else is reconstructed from these CLI args. Mismatched flags load 80/80 tensors silently but run the WRONG forward pass — false-positive scores. Cross-check with the `mythic_rdt_config.json` saved next to the ckpt before authoring the eval command.
- **[pattern:project]** Belt-and-suspenders pod backup: after long training runs, mirror artifacts to solidPC via TWO redundant cron paths — (1) Claude session `CronCreate` (in-memory, dies with session, fires every 25 min off-the-hour like `17,42 * * * *`), AND (2) system crontab via `(crontab -l; echo '*/25 * * * * /bin/bash <sync_script>') | crontab -` (durable across reboots, runs even if Claude isn't open). The two windows never align so coverage is continuous. Sync script uses `rsync -az --partial -e 'ssh -p <port> -o StrictHostKeyChecking=no -o ConnectTimeout=15 -o ServerAliveInterval=10'` — `--partial` preserves bytes on transient disconnects, ConnectTimeout/KeepAlive prevent hung SSH.
- **[convention:project]** Add `pod_wandb/` to `.gitignore` alongside `wandb/`. The repo mirrors pod-side wandb runs locally for backup; without the gitignore line, dozens of `run-*` directories (each ~1MB of metrics + symlinks) end up staged on every commit and pollute `git status`.
- **[gotcha:project]** `git add <file>` only stages the one file you named, but `git status` still shows other dirty files in the worktree as ` M` (worktree-modified, not staged). Confirm what's actually going INTO a commit via `git diff --cached --stat` (or `git status --short` and look for `M ` / `A ` in column 1 vs ` M` in column 2) — don't assume the commit will only include what you `git add`-ed if you also have unstaged changes elsewhere. The Caliber pre-commit hook only refreshes already-staged files; it won't accidentally stage the worktree-dirty ones.
